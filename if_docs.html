IF-Space Engine Documentation

Description

The IF-Space Engine Software is designed to make creating works of interactive fiction easy, fun, and code free. Three styles of interactive fiction are available for creation with this software:

  1. Classic
    • Games which require the user to input strings of text to further the gameplay (like Colossal Cave Adventure and Zork)
    • Utilizes: Entire range of node tools
  2. Modern
    • Games which generate buttons for the possible in-game actions as well as the possible directions in which to move
    • Utilizes: Node Tools Header, Directions, Descriptions, Actions, Win, Lose
  3. Gamebook
    • Games which only generate buttons for the possible directions in which to move (Like a choose your path gamebook)
    • Utilizes: Node Tools Header, Directions, Descriptions, Win, Lose

Layout

Node Map

- Left hand side of the page, consisting of the grid space

Node Tools

- Right hand side of the page, consisting of the various tabs and options

Game Sim

Functions

Node Map

The grid space on the left side of the software is known as the 'Node Map'. It is here where the "map" of your game is realized.

The red colored square in the Node Map indicates the Node which you currently have selected and which would be edited if changes are made in the Node Tools.

Blue colored squares indicate nodes which exist but that are not currently active in the Node Tools. To switch from the currently active node (the red one) to an inactive node, simply click whichever inactive node you would like to edit.

Black colored squares (empty squares) are locations which are available to create a node. To do this, simply click inside of an empty square which you would like to be a node, and then click "Ok" when you are asked if you are sure.

To delete any given node and all of its contents, first select and make active the node that is to be deleted, and then click the "Delete Node" button in the top left. You will then be prompted to click "Ok" or "Cancel" to make sure this was not a mistake.

The zoom in and out functions can be found in the bottom bar of the software. Clicking these will either enlarge or shrink the grid space, respective to the button you click.

The z-index input is responsible for changing the z coordinate of the Node Map. Although the Node Map is represented in a standard X,Y coordinate plane, the Z axis still exists and can be changed with the z-index input. This is useful for games where the map is meant to maintaine the true form of the available directions and which contain Up or Down directions (which default to increasing or decreasing the z coordinate, respectively). This particular feature can be ignored, but to ignore this feature while also including Up or Down directions in the game will require changing the destination location of those directions in the Directions tab (see Directions tab below for more information).

Node Tools

Node Tools Header

The Node Tools Header can be thought of as the basic setup for each node. It consists of the Location of the node, the given Name of the node, the default Visibility of the node, and the number of Points available for visiting this node.

  • Location
    • The constant X,Y,Z coordinate of the currently selected node. This cannot be changed.
  • Name
    • This field is for the Name of the current node. This will be printed to the screen during gameplay each time the user visits this node.
  • Visibility
    • Default value of this node's visibility
    • Determines whether or not the potential items in this node will be available for pickup and printed onscreen when visiting the node and/or using the "Look" command.
    • Checked means the items will be available, and unchecked means they will not be available or printed to screen
  • Points
    • The number of points the user will receive upon visiting this node for the first time.

Directions Tab

The Directions Tab is used to define the various directions in which a player can move from this node during gameplay.

  • Checkbox Directions
    • By checking one of these boxes, you are adding the potential for a player to go in this direction during gameplay.
    • After checking a box, the directions's additional options are displayed below the checkboxes.
    • Unchecking a box will remove the potential for that direction during gameplay and will also erase any of the options for that direction that were changed while the box was checked.
    • NOTE: The node in which this direction leads to is not automatically created (since the destination location can be edited). You must create the node that this direction leads to manually in the Node Map.
  • Location Setting
    • The Location Setting is the node location in which the player will be sent when successfully navigating in this direction during gameplay.
    • The default Location Setting is the location on the Node Map which would naturally occur when moving from the currently selected node (the node being edited) one position in the checked direction being edited.
    • This Location Setting can be changed to any node (in the form of x,y,z).
  • Alternatives Setting
    • Alternative names given to the checked directions which would also result in the player moving in the checked direction. For example: On top of saying 'Go North', the user may naturally want to say 'Follow path' or 'Take path'. The comma separated list in the 'Alternatives' field makes up all of the other potential means of going in the checked direction.
  • Exclude Defaults Setting
    • Checkbox for removing the possibility of using the default "Go {direction}" command. For example, if the "North" direction is checked, and the 'Exclude Defaults' checkbox is checked, then the user cannot type "Go North" to go this direction; they must use one of the listed 'Alternatives' instead.
    • By default, this checkbox is unchecked - therefore including the "Go {direction}" command in the available commands for moving in this direction.
    • This setting is especially useful for Modern and Gamebook styles, since a singular Alternative to 'Go {direction}' is likely desired.
  • Requirements
    • The requirements for this direction to be available for use by the player during gameplay
    • See the 'Requirements' section below for more details on 'Requirements'.

Descriptions Tab

The Descriptions Tab is where a node's initial text (upon visiting the node in gameplay) is defined.

  • Default Setting
      • This is the text which will be printed during gameplay the first time the user visits this node. Typically it would include any continued plot which is being told as well as any setting features which should be described to the player.
  • Basic Setting
    • This is the text which will be printed during gameplay upon subsequent visits to this node as well as after the "Look" command. Typically this includes any setting features which should be described to the player.
  • Evolutions
    • Evolution descriptions are printed during gameplay (in place of default and basic descriptions) when their requirements are met.
    • A node may have any number of evolution descriptions, including zero.
    • If many evolutions exist, the right most evolution description which meets its requirements will be the description that is printed to the screen when a player visits this node.
    • See the 'Requirements' section below for more details on 'Requirements'.

Items Tab

The items tab is used to create and define the items which will start in this node. Items are things in the game that the player can pick up, look at, and drop anywhere in the game. Players can check which items are on their person by using the "Inventory" command.

  • Add Item Button
    • Adds an item to the list of items which would begin in this node.
  • Remove Item Button
    • Removes the item related to this particular button from the item list associated with this node.
  • Name Setting
    • This input is for a list of all the different possible names referring to this item. For example: Flashlight, torch, light
    • The first name in the list will be used for the onscreen referral to this item.
    • Should be in the form of a comma separated list
  • Description Setting
    • This description will be the output for when the player uses the "Look {item}" command.
  • Points
    • The number of points the user will receive for having this item in their inventory.
  • Requirements
    • The requirements for this item to be available for pickup by the player during gameplay
    • See the 'Requirements' section below for more details on 'Requirements'.
  • Add Evolution Button
    • Adds an evolution to this item
    • Item evolutions can be used to change the description of an item given its set of requirements are satisified.
    • Any number of evolutions can be added to an item.
    • If multiple item evolutions exist, the bottom most evolution that has its requirements satisfied will have its description used for the "Look {item}" command.
  • Remove Evolution Button
    • Removes the corresponding item evolution from the list of evolutions for this item

Containers Tab

The containers tab is used to define containers which can be found in this node. A container is something that can hold and store items. Containers in IF-Space can be configured so that having certain items or a certain number of items in a particular container can trigger requirements for other settings in the game (similar to filling the trophy case in Zork).

  • Add Container Button
    • Adds a container to the list of containers for this node
  • Remove Container Button
    • Removes the corresponding container from the list of containers for this node
  • Name Setting
    • This input is for a list of all the different possible names referring to this container. For example: Bookshelf, Bookcase, Shelf
    • The first name in the list will be used for the onscreen referral to this container.
    • Should be in the form of a comma separated list
  • Capacity Setting
    • This setting defines the number of items that can be held by the container at one time.
    • Defaults to infinity
  • Completed Contents Setting
    • This field describes what must be stored in the container in order for this container's requirement check to be satisfied.
    • This field accepts 1 of 2 types of input:
      1. A comma separated list of items (E.g. key, cash, note)
      2. A number (E.g. 5)
    • If a list of items is provided, each of those items must be stored in the container for its requirement check to be satisfied.
    • If a number is provided, a minimum of this many items must be stored in the container for its requirement check to be satisfied.
  • Illegal Items Setting
    • A comma separated list of items (which exist in the game) that cannot be stored in this container
    • The first name in an item's list of potential names must be the name entered in the illegal items list.
  • Points Setting
    • The number of points the user will receive for having this container's "Completed Contents" requirement satisfied
  • Requirements
    • The requirements for this container to be available for use by the player during gameplay
    • See the 'Requirements' section below for more details on 'Requirements'.

Actions Tab

The Actions Tab is used to create and define the various possible actions that the player can potentially perform in this node. For example: "Break window" or "Open locker"

  • Add Action Button
    • Adds an action to the list of actions for this node
  • Remove Action Button
    • Removes the corresponding action from the list of actions for this node.
  • Action(s) Setting
    • This input is for a list of all the different possible action calls referring to this action. For example: Break window, Smash window, Destroy window
    • Should be in the form of a comma separated list
  • Max Uses Setting
    • The maximum number of times that this action can be used
    • Defaults to infinity
  • Costs Setting
    • A comma separated list of items which are removed from the player's inventory and effectively vanish after performing this action
    • The first name in an item's list of potential names must be the name entered in the "Costs Setting" list.
  • Drops Setting
    • A comma separated list of items which are dropped from the player's inventory after performing this action (E.g. Throwing a rock should probably drop a rock)
    • The first name in an item's list of potential names must be the name entered in the "Drops Setting" list.
  • Visibility Setting
    • This selection determines how the visibility setting of this node will change after performing this action.
    • No Change keeps the visibility setting how it already is
    • On and Off sets the visibility to On and Off, respectively
    • Switch changes the visibility to the opposite of the current setting (i.e. On goes to Off and Off goes to On)
  • Action Response Setting
    • The Action Response is the text that will be printed to the screen during gameplay after successfully using an action.
    • For example: If the action is "Throw Rock" then the Action Response could be "The rock breaks the window and falls to the ground."
  • Fail Response Setting
    • The Fail Response is the text that will be printed to the screen during gameplay after trying to call an action which has failing requirements.
  • Points Setting
    • The number of points the user will receive for successfully calling this action
  • Requirements
    • The requirements for this action to be available for use by the player during gameplay
    • See the 'Requirements' section below for more details on 'Requirements'.

Win Tab

The Win Tab is for defining a node as having the potential for winning the game in this node. If the Win Description is not empty, then it is assumed that the node has a win condition. If there is a win condition and the requirements in the Win Tab are satisified, then the player wins the game.

  • Win Description
    • This is the text that will be printed to the screen during gameplay when the player wins the game.
  • Requirements
    • The requirements for winning the game in this node
    • See the 'Requirements' section below for more details on 'Requirements'.

Lose Tab

The Lose Tab is for defining a node as having the potential for losing the game in this node. If the Lose Description is not empty, then it is assumed that the node has a lose condition. If there is a lose condition and the requirements in the Lose Tab are satisified, then the player loses the game.

  • Lose Description
    • This is the text that will be printed to the screen during gameplay when the player loses the game.
  • Requirements
    • The requirements for losing the game in this node
    • See the 'Requirements' section below for more details on 'Requirements'.

Hint Tab

The hint tab is used to offer the player a hint in the event that they use the "Help" command or if their input is unrecognized 5 consecutive times.

  • Leave the hint field empty if this node should have no hints.

Requirements

The requirements section exists for many of the features in the IF-Space Engine. They are a powerful tool for making logical changes across these features. In general, a particular setting or feature that has requirements attached to it will only become available for use during gameplay if those reuirements are satisified, otherwise this feature will be absent from the game until those requirements are met.

  • Items Requirement
    • A comma separated list of items needed in the player's inventory
    • The first name in an item's list of potential names must be the name entered in the "Items Requirement" list.
  • Containers Requirement
    • A comma separated list of containers required to have their "Completed Contents" setting satisfied
    • The first name in the container's list of potential names must be the name entered in the "Containers Requirement" list.
  • Local Actions Requirement
    • A comma separated list of actions required to have been successfully called within this node
    • The first name in the action's list of potential names must be the name entered in the "Local Actions Requirement" list.
  • Global Actions Requirement
    • A comma separated list of actions required to have been successfully called within any existing node
    • The first name in the action's list of potential names must be the name entered in the "Global Actions Requirement" list.
  • Previous Action Requirement
    • A singular action which is required to have been the most recent successfully called action
    • The first name in the action's list of potential names must be the name entered in the "Previous Action Requirement".
  • Item Evos Requirement
    • A comma separate list of brackets which contain a singular required item and a number representing that item's required evolution stage
    • For example: [key, 1], [gemstone, 2]
      • This example would require that the player has in their inventory a key with a first stage evolution as well as a gemstone with its second stage evolution

Node Tools Footer

  • Game Settings
    • This button opens the Game Settings popup.
      • Game Style
        • This defines the game's game style (as described in the IF-Space Engine description above)
      • Game Rating
        • This dropdown assigns a maturity rating to your game.
      • Game Author
        • This field is for the name of the creator of the game (to be displayed onscreen at the beginning of gameplay).
  • Game Title Setting
    • This defines the name of the game and will print to screen in large text at the start of gameplay
  • Load Game
    • For loading a previously saved game
  • Save Game
    • For saving the current game
